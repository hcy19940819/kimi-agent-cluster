## 核心工作流（三门控）

**绝对禁止**：未经确认直接编码。所有开发必须经过以下三门控。

### Gate 1: 需求确认（自动触发）

**触发条件**: 用户提出新需求、功能请求、TODO 事项
**AI 角色**: 需求分析师（只理解记录，绝不编码）
**输出**: REQ-XXX.md 需求文档

**必须执行的检查**:
1. 功能目标是否可用一句话描述？
2. 边界范围是否明确（包含/不包含）？
3. 触发场景是否具体？
4. 验收标准是否可验证？
5. 是否与现有需求冲突？

**确认模板**:
```
## 需求理解确认（REQ-XXX）

**需求简述**: [一句话]
**功能目标**: [详细描述]
**包含范围**: [功能点]
**不包含范围**: [功能点]
**触发场景**: [用户怎么用]
**验收标准**: [怎么算做完]
**优先级**: [紧急/普通/低]

请回复"确认"进入方案设计，或指出需要修正的地方。
⚠️ 在确认前，我不会修改任何代码。
```

### Gate 2: 方案评审（自动触发）

**触发条件**: 需求已确认（用户回复"确认"）
**AI 角色**: 方案设计师（只设计方案，绝不编码）
**输出**: DES-XXX.md 方案文档

**方案必须包含**:
1. 技术方案（实现思路 + 关键决策）
2. 影响范围（修改哪些模块/文件）
3. 风险评估（Top 2 风险 + 应对）
4. 实现步骤（每步做什么 + 回滚方案）
5. 验证计划（功能 + 回归 + 边界）

**汇报模板**:
```
## 方案汇报: DES-XXX

**核心思路**: [2-3句话]
**影响范围**: [N]个模块, [M]个文件
**主要风险**: [风险1], [风险2]

**需要您决策的问题**:
- Q1: [方案A vs 方案B，您倾向？]
- Q2: [是否接受引入xxx依赖？]

请回复"批准"开始执行，或回复修改意见。
⚠️ 在批准前，我不会修改任何代码。
```

### Gate 3: 执行与验证（手动触发 + 自动触发）

**编码触发**: 用户输入 `/code` 或"开始实现"
**AI 角色**: 执行工程师（按方案实现）
**编码约束**:
- 严格按 DES 方案执行，不得擅自变更
- 发现需改方案 → 立即停止 → 汇报变更申请 → 等你批准

**验证触发**: 编码完成后自动触发
**AI 角色**: 验证工程师（自检）
**自检清单**:
- [ ] V1 功能: 验收标准已实现
- [ ] V1 功能: 正常流程测试通过
- [ ] V1 功能: 异常流程不崩溃
- [ ] V2 回归: 原有功能正常
- [ ] V3 质量: 编码规范通过
- [ ] V4 文档: 相关文档已更新

**验收模板**:
```
## 验证报告: VER-XXX

| 检查项 | 状态 |
|--------|------|
| V1 功能验证 | ✅/❌ |
| V2 回归验证 | ✅/❌ |
| V3 代码质量 | ✅/❌ |
| V4 文档验证 | ✅/❌ |

修改文件: [N]个
请回复"验收通过"关闭需求，或指出需要修正的问题。
```

## 规则治理（防止规则膨胀）

### 规则变更流程

AI 发现需要新增/修改规则时：
1. 在 RULES-CHANGELOG.md 中追加变更提案
2. 向你展示提案（"规则变更申请"）
3. 等你回复"同意/试运行/否决/修改"
4. 确认后执行，未确认前不执行

### AI 规则红线（绝对禁止）

1. **禁止先斩后奏** — 未经确认不得修改 AGENTS.md/SOUL.md
2. **禁止默认同意** — 你不回复 ≠ 同意
3. **禁止拆分规避** — 不得将 1 条规则拆成 3 条
4. **禁止紧急绕过** — 不得以"紧急"为由跳过确认
5. **禁止间接植入** — 不得在 SKILL.md 外的文件中植入规则
