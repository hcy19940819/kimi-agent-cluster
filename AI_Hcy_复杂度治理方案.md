# AI_Hcy 复杂度治理方案

> 诊断日期: 2026-04-29
> 问题: 规范复杂度失控，规则遗漏频发
> 方案目标: 84KB/82条规则 → ~20KB/20条规则，消除遗漏

---

## 第一部分: 现状诊断

### 一、关键数字

| 指标 | 数值 | 健康阈值 | 状态 |
|------|------|---------|------|
| 规范总量 | 84.5KB | ≤30KB | 🔴 超标181% |
| 规则总数 | 82条 | ≤30条 | 🔴 超标173% |
| 规范文件数 | 12个 | ≤5个 | 🔴 超标140% |
| P0事故文件占比 | 43.3% | ≤20% | 🔴 超标117% |
| 信息冗余率 | 20.2% | ≤10% | 🟡 超标102% |
| 规则重复组数 | 10组 | ≤3组 | 🔴 超标233% |
| 规则冲突+不可执行 | 15.9% | ≤5% | 🔴 超标218% |
| 协议嵌套深度 | 4层 | ≤2层 | 🔴 超标100% |
| **综合评分** | **43.4/100** | **≥60** | **🟡 中度风险** |

### 二、核心问题确认

**你的感受完全正确。规则遗漏不是执行问题，而是架构问题。**

当前陷入了恶性循环:

```
规则膨胀(82条) → 认知超负荷 → 规则遗漏 → 问题发生 → 新P0文件(+7KB) → 新规则(+3条) → 规则更膨胀
```

3起P0事故产生了5个新文件(36.6KB)，规则从AGENTS.md的9条膨胀到82条(+811%)，但问题还在发生(#48/#49/#51)。

### 三、五层根因

| 层级 | 问题 | 证据 |
|------|------|------|
| 规则过载 | 82条超过认知上限 | 有效规则仅占63.9% |
| 机制缺陷 | 规则"只增不减" | 8条经验→19条规则，无淘汰机制 |
| 架构冲突 | 遵守A则违反B | 5组冲突+2组行为矛盾 |
| 结构过载 | 单文件失控 | TODO.md 20KB, MEMORY.md 15KB |
| 文化惯性 | "新文件=新规范" | 3起P0→5个文件，碎片化 |

### 四、关键发现

**发现1: 规则自相矛盾**

- AGENTS.md要求"主动识别skill"(看到关键词自动判断)
- MEMORY.md要求"禁止臆测需求"(所有判断需用户确认)
- **AI被迫在"自主判断"和"等待指令"间随机选择** → 表现为规则遗漏

**发现2: 文件越不敢改越大**

- MEMORY.md 15KB、TODO.md 20KB → 想加规则不敢改 → 新建P0文件 → 更碎片化

**发现3: 紧急规则被淹没**

- AGENTS.md的3条红线 → 膨胀到22条衍生规则 → 紧急规则沉在文档底部 → 记不住

## 第二部分: 新架构方案

### 一、核心设计哲学

> **规则越少越好，执行越简单越好。**
>
> 用"更少但更强的约束 + 自动化"打破恶性循环。

**五大设计原则:**

| 原则 | 含义 | 效果 |
|------|------|------|
| 元规则优先 | 先定义"如何管理规则"，再定义具体规则 | 防止规则无限增长 |
| 单一来源 | 一个规则只存在于一个地方 | 消除重复和冲突 |
| 场景驱动 | 按使用场景(启动/开发/提交/事故)组织 | 需要时才加载 |
| 强制自动化 | 能脚本化的绝不靠人记 | 让人不知道规则就能遵守 |
| 约束有界 | 核心规则≤15条，新增必须淘汰旧规则 | 硬性上限 |

### 二、新文件结构 (8个文件)

```
AI_Hcy/
├── README.md              # 入口导航（仅链接，无规则）
├── META.md                # ★ 元规则 + 执行规则（唯一可信来源，~2KB）
├── CONTEXT.md             # 项目快照（<1KB，每次会话更新）
├── SKILLS.md              # 技能索引表（<2KB）
├── playbook/              # 场景手册（按场景分，不是按文件类型）
│   ├── start.md           # 会话启动（3步替代旧5步）
│   ├── develop.md         # 开发工作流
│   ├── submit.md          # 提交验证
│   └── incident.md        # 事故处理（所有P0收敛于此）
└── scripts/               # 自动化脚本
    ├── pre-flight.sh      # 启动自检
    ├── verify.sh          # 提交前验证
    └── guard-rail.py      # 运行时守护
```

**旧文件处置:**

| 旧文件 | 处置 | 内容去向 |
|--------|------|---------|
| AGENTS.md | **删除** | 3条红线→META.md，协议→脚本 |

| TODO.md | **拆分** | 活跃项≤50保留，历史归档到archive/ |

| .skills/todo-manager/ | **重构** | 合并为SKILLS.md索引 |

### 三、三层治理架构

#### Layer 1: 元规则层 (META.md)

**5条元规则 — 永远不变，是"规则的宪法":**

```
MR1. 单一来源
     一个规则只存在于一个地方。发现重复立即删除最旧的那份。

MR2. 场景驱动
     规则按使用场景(启动/开发/提交/事故)组织，不按文件类型组织。

MR3. 自动化优先
     能脚本化的规则绝不靠人记。脚本未覆盖的规则视为无效。

MR4. 约束有界
     核心执行规则 ≤ 15条。新增必须淘汰旧规则。

MR5. 经验 ≠ 规则
     经验教训只能在incident.md中记录，不能自动升级为规则。
     升级需经过: 提案 → 三问检查 → 试运行2周 → 审批。
```

**15条执行规则:**

```
【自动化层 — 脚本覆盖，无需记忆】
R1. 会话启动 → 运行 pre-flight.sh
R2. 代码提交 → 运行 verify.sh
R3. 死循环检测 → 同一操作>3次自动终止

【决策约束层 — 3条红线】
R4. 不改需求外内容
R5. 执行前确认
R6. 文档与代码同步提交

【质量约束层】
R7. 同一Shell命令最多重试3次
R8. 确认对话最多2轮，未决上报
R9. 遗留问题必须标记，禁止静默跳过
R10. 需求必须标明来源[user/tech-debt/incident#N/assumed]

【信息管理层】
R11. 一个规则只存在于一个地方
R12. 当前任务上下文写在CONTEXT.md
R13. 事故记录写入incident.md，禁止新文件

【元规则约束层】
R14. 新增规则必须淘汰旧规则(总数≤15)
R15. 经验教训≠规则，升级需审批
```

**压缩效果: 60+条 → 15条执行规则 + 5条元规则 = 75%缩减**

#### Layer 2: 上下文层 (CONTEXT.md + SKILLS.md)

**CONTEXT.md** 是"我是谁、现在在哪"的快照:

```markdown

# Session（每次会话更新）

pytest tests/ -x -q

```

**目标: <1KB，只保留当前必须知道的。**

#### Layer 3: 场景手册层 (playbook/*.md)

**按场景分4个文件，每个≤30行:**

| 手册 | 场景 | 核心内容 |
|------|------|---------|
| start.md | 每次会话开始时 | 3步: 读CONTEXT→读META→确认目标 |
| develop.md | 写代码时 | 需求确认→分层开发→测试→文档同步 |
| submit.md | commit前 | 运行verify.sh→通过→提交 |
| incident.md | 出问题时 | 记录→分析→修复→验证→关闭 |

**incident.md 替代所有P0_*.md:**

### 四、防呆机制设计

> **最好的规则是让人不知道规则的存在就能遵守。**

| 防呆级别 | 机制 | 覆盖规则 |
|---------|------|---------|
| **强制层** | 脚本拦截，不通过无法继续 | R1启动自检、R2提交验证、R3死循环检测 |
| **默认层** | 默认行为就是正确行为 | R4不改需求外、R5执行前确认、R6文档同步 |
| **检测层** | 违规自动检测并告警 | R7命令≤3次、R8确认≤2轮、R9遗留标记、R10需求来源 |
| **流程层** | 流程设计使违规更困难 | R14约束有界、R15经验≠规则 |

**3个脚本覆盖8条规则:**

| 脚本 | 覆盖规则 | 功能 |
|------|---------|------|
| pre-flight.sh | R1, R11, R12 | 启动时检查META.md存在、CONTEXT.md存在、文件数≤8 |
| verify.sh | R2, R6, R9, R10, R13 | 提交前检查文档同步、TODO标记、需求来源、禁止新建P0文件 |
| guard-rail.py | R3, R7, R8 | 运行时检测死循环、命令重试次数、确认轮数 |

### 五、规则生命周期管理

**新增规则的"四道闸门":**

```
经验库(incident.md) 
    → "规则化提案" 
    → 三问检查(是重复问题?能脚本化?淘汰哪条旧规则?)
    → 试运行2周
    → 转正(入META.md)或废弃
```

**定期Review(每30天):**

- [ ] 当前执行规则数量 ≤ 15?
- [ ] 每条规则过去30天是否被触发过?(未触发→删除)
- [ ] 是否有规则被脚本覆盖?(被覆盖→删除)
- [ ] 是否有互相矛盾的规则?(有→修正)

**自动淘汰条件(满足任一即触发):**

1. 连续30天未被引用
2. 已被脚本自动化覆盖
3. 与其他规则矛盾
4. 新增规则导致总数>15

## 第三部分: 迁移计划

### 六阶段渐进迁移 (预计4周)

采用**"桥梁并行"模式**: 前4阶段只增不减旧文件，双轨运行；第5阶段切换为主；第6阶段清理旧文件。

| 阶段 | 名称 | 耗时 | 核心动作 | 关键交付物 |
|------|------|------|----------|-----------|
| **Phase 1** | 准备与审计 | 2-3天 | 完整备份、生成校验和、创建BRIDGE.md桥梁文件 | 仓库快照、文件清单、映射表 |
| **Phase 2** | 奠基(L1+L2) | 3-4天 | 创建META.md、CONTEXT.md、目录结构 | 元规则、项目快照、playbooks/目录 |
| **Phase 3** | 核心(TODO拆分+Playbooks) | 5-7天 | TODO拆分为活跃(≤50项)+归档、创建4个场景手册 | 精简TODO、4个playbook、verify.sh |
| **Phase 4** | 整合(P0合并+验证) | 4-5天 | 5个P0文件整合为incident.md、完整性验证 | incident-log、回归测试通过 |
| **Phase 5** | 切换(新架构为主) | 3-5天 | 旧文件加"已迁移"标记、5天监控期 | BRIDGE.md更新、监控报告 |
| **Phase 6** | 清理(旧文件归档) | 2-3天 | 旧文件移入archive/legacy/、删除桥梁文件 | 归档完成、CHANGELOG更新 |

**回滚机制:**

| 回滚级别 | 触发条件 | 操作 | 耗时 |
|---------|---------|------|------|
| 阶段内回滚 | 当前阶段发现问题 | 删除本阶段新增文件 | <5分钟 |
| 跨阶段回滚 | 后阶段发现严重问题 | 回到上一阶段的备份 | <15分钟 |
| 完全回滚 | 新架构整体不可用 | 用Phase 1备份还原整个仓库 | <30分钟 |

### 关键风险与缓解

| 风险 | 概率 | 缓解措施 |
|------|------|---------|
| TODO拆分遗漏关键项 | 中 | 拆分脚本生成索引表 + 人工抽检20% |
| P0整合丢失细节 | 中 | 逐文件人工审核 + 原文件保留不删除 |
| 切换后AI找不到规则 | 中 | 5天监控期 + BRIDGE.md引导 + 快速回滚 |
| 新规则15条不够用 | 低 | MR4允许淘汰旧规则，保持动态平衡 |

## 第四部分: 预期效果

### 量化指标对比

| 指标 | 旧架构 | 新架构 | 改善 |
|------|--------|--------|------|
| 核心规则数量 | 60+条 | ≤15条 | -75% |
| 规范文件总数 | 12个 | 8个 | -33% |
| 规范总大小 | 84.5KB | ~20KB | -76% |
| MEMORY.md | 15KB | <1KB (CONTEXT.md) | -93% |
| TODO.md | 20KB | ≤50活跃项+归档 | -75% |
| P0事故文件 | 5个且持续增长 | 1个 (incident.md) | 收敛 |
| 启动协议步骤 | 5步 | 3步 | -40% |
| 规则冗余率 | 20.2% | ~5% | -75% |
| 脚本自动化覆盖率 | ~10% | ~53% (8/15条) | +430% |
| 预计规则遵守率 | ~50% | >80% | +60% |

### 定性改善

1. **AI不再"记不住规则"**: 15条执行规则 vs 82条，认知负荷降低82%
2. **不再自相矛盾**: 单一来源消除冲突，"主动识别skill"vs"禁止臆测"的矛盾通过R10(标来源)解决
3. **经验不再自动变规则**: MR5+四道闸门，防止恶性循环
4. **P0不再产生新文件**: 所有事故收敛到incident.md，防止碎片化
5. **遵守规则比不遵守更容易**: 3个脚本自动拦截，默认行为即正确行为
