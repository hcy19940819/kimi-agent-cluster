## 二、Kimi 2.6 最佳实践

### 2.1 Prompt 写法：怎么写效果最好

#### 原则1：结构化 > 自由文本

K2.6对结构化的指令响应最好。**不要这样写**：

```
帮我改一下这个登录功能
```

**要这样写**：

```markdown
## 任务
优化用户登录模块的性能

## 当前代码文件
- app/auth/routes.py（登录路由）
- app/models/user.py（用户模型）
- app/utils/jwt.py（JWT工具）

## 具体要求
1. 减少数据库查询次数（当前每次登录查2次，希望优化到1次）
2. 添加登录失败次数限制（5次锁定15分钟）
3. 保持现有API接口不变

## 输出格式
- 只输出需要修改的文件内容
- 每个修改标注[性能优化]、[安全增强]或[保持兼容]
- 不需要解释修改原因
```

#### 原则2：让模型扮演特定角色

在system prompt或对话开头明确角色设定，K2.6的角色遵循能力非常强：

```markdown
你是一位有10年经验的Python后端架构师，专注于Flask应用的高性能优化。
你的特点是：
- 优先考虑数据库查询效率
- 注重安全最佳实践
- 代码简洁不冗余
- 所有建议必须附带具体的代码实现，不只是思路
```

#### 原则3：使用XML/JSON标签包裹代码

当传入多文件代码时，用明确的标签包裹，K2.6的上下文定位会更准确：

```xml
<file path="app/auth/routes.py">
@app.route('/login', methods=['POST'])
def login():
    # ... 代码内容
</file>

<file path="app/models/user.py">
class User(db.Model):
    # ... 代码内容
</file>
```

#### 原则4：分步骤下达指令（利用长上下文优势）

K2.6的256K上下文允许你在一个对话中积累大量背景，然后基于这些背景下达精确指令：

**第一轮**（建立上下文）：
```
我会把视频管家项目的核心代码传给你，请理解项目结构和逻辑关系。
先传 app/__init__.py ... [代码]
```

**第二轮**（补充上下文）：
```
继续传app/routes/下的路由文件 ... [代码]
```

**第三轮**（基于已积累的上下文下达任务）：
```
基于前面传的所有代码，现在需要添加一个新功能：
视频上传后自动转码。请在以下位置实现：
1. app/services/ 下新增transcode.py
2. app/models/video.py 添加转码状态字段
3. app/routes/video.py 添加上传回调接口
```

> **技巧**: 前缀缓存机制让这种多轮对话的成本极低——第一次完整计费，后续只要prompt前缀一致，缓存命中部分只收约25%费用。

### 2.2 长上下文的高效利用

#### 策略1：项目代码地图法

把一个文件的"摘要"而非"完整内容"放入上下文，让K2.6知道有什么，需要时再展开：

```markdown
## 项目结构地图

### 模型层 (app/models/)
| 文件 | 职责 | 关键类 |
|------|------|--------|
| user.py | 用户管理 | User(id, username, email) |
| video.py | 视频管理 | Video(id, title, file_path, status) |
| category.py | 分类管理 | Category(id, name, parent_id) |

### 路由层 (app/routes/)
| 文件 | 职责 | 关键端点 |
|------|------|----------|
| auth.py | 认证相关 | /login, /register, /logout |
| video.py | 视频CRUD | /videos, /videos/<id>, /upload |

### 服务层 (app/services/)
| 文件 | 职责 |
|------|------|
| upload.py | 文件上传处理 |
| transcode.py | 视频转码（待实现） |
```

这种方式在256K上下文中可以容纳数十个文件的摘要，K2.6能基于地图理解全局，再针对具体文件深入。

#### 策略2：差异化上下文加载

不同类型的任务需要不同的上下文策略：

| 任务类型 | 推荐上下文策略 | 原因 |
|----------|---------------|------|
| 新增功能 | 加载项目地图 + 相关模型/路由文件 + 类似功能参考 | 需要理解现有模式 |
| Bug修复 | 加载错误堆栈 + 相关文件 + 最近修改记录 | 聚焦问题范围 |
| 代码重构 | 加载完整待重构模块 + 测试文件 | 确保重构后测试通过 |
| 代码审查 | 加载变更文件 + 相关依赖文件 | 理解变更影响范围 |

#### 策略3：使用"前缀续写"强制输出格式

K2.6支持OpenAI风格的"前缀续写"，可以强制JSON输出：

```python
response = client.chat.completions.create(
    model="kimi-k2.6",
    messages=[
        {"role": "system", "content": "你是一个API设计助手，只输出JSON格式的接口文档"},
        {"role": "user", "content": "为视频转码功能设计API接口"},
        {"role": "assistant", "content": '{"api_name": "视频转码", "endpoints": ['}  # 强制从这儿继续
    ]
)
```

### 2.3 Agent / Function Calling 能力使用

#### K2.6的Agent能力升级点

相比K2.5，K2.6在Agent方面的关键升级：

1. **子代理数量**: 100个 → 300个（你的多Agent系统可以更大规模）
2. **协调步骤**: 1500步 → 4000步（长流程更稳定）
3. **持续运行**: 支持数天不间断（监控/运维类Agent可以长期驻留）
4. **工具调用准确率**: 显著提升（减少"调错工具"的情况）

#### Function Calling最佳实践

**定义工具时，docstring就是prompt**：

```python
def search_videos(
    keyword: str,
    category: str = None,
    date_range: str = None,
    sort_by: str = "relevance"
) -> str:
    """
    搜索视频库中的视频内容。

    - 用户输入关键词查找视频
    - 需要按分类筛选视频
    - 需要查看特定时间段上传的视频
    
    参数说明：
    - keyword: 搜索关键词，支持视频标题和描述模糊匹配
    - category: 分类名称，可选值：["电影", "电视剧", "综艺", "纪录片", "动画"]
    - date_range: 时间范围，格式 "2025-01-01,2025-06-30"
    - sort_by: 排序方式，可选 relevance(相关度)/time(时间)/views(播放量)
    
    返回: JSON格式的视频列表
    """
    # 实现代码...
```

> **关键**: 工具描述要写清楚"什么场景用"，而不只是"这个工具做什么"。K2.6会根据场景匹配工具，描述越具体，匹配越精准。

**注册工具的标准模式**：

```python
import json
from openai import OpenAI

client = OpenAI(api_key="YOUR_KEY", base_url="https://api.moonshot.cn/v1")

# 1. 定义工具描述（OpenAI兼容格式）
tools = [
    {
        "type": "function",
        "function": {
            "name": "search_videos",
            "description": "搜索视频库，适用于用户按关键词/分类/时间查找视频",
            "parameters": {
                "type": "object",
                "properties": {
                    "keyword": {"type": "string", "description": "搜索关键词"},
                    "category": {"type": "string", "enum": ["电影", "电视剧", "综艺", "纪录片", "动画"]},

# 2. 调用模型（启用工具）
```

### 2.4 文件上传与上下文管理技巧

#### 批量文件处理的推荐流程

```python
# 步骤1: 读取项目文件并生成摘要（预处理）
import os

def scan_project(project_path):
    """扫描项目结构，生成代码地图"""
    file_map = []
    for root, dirs, files in os.walk(project_path):
        # 跳过不必要的目录
        dirs[:] = [d for d in dirs if d not in ['__pycache__', '.git', 'venv', 'node_modules']]
        for file in files:
            if file.endswith(('.py', '.js', '.html', '.css')):
                filepath = os.path.join(root, file)
                with open(filepath, 'r', encoding='utf-8') as f:
                    content = f.read()
                # 只保留前50行作为摘要
                summary = '\n'.join(content.split('\n')[:50])
                file_map.append({
                    'path': filepath,
                    'summary': summary,
                    'size': len(content)
                })
    return file_map

# 步骤2: 将代码地图发送给K2.6，建立全局上下文
# 步骤3: 针对具体任务，只加载相关完整文件
```

#### 上下文分段策略

当项目太大（超过256K），使用分段加载：

| 分段方式 | 适用场景 | 操作方法 |
|----------|----------|----------|
| 按模块分段 | 大型项目 | 每次加载一个业务模块的完整代码 |
| 按层次分段 | 清晰的分层架构 | 先传模型层，再传服务层，最后传路由层 |
| 增量更新 | 持续迭代 | 只传变更文件+相关依赖 |

### 2.5 K2.6 已知限制与规避方法

| 限制 | 影响 | 规避方法 |
|------|------|----------|
| **前缀缓存TTL 5分钟** | 低频对话缓存失效，费用上升 | 批量处理任务集中完成，或用脚本保持心跳 |
| **Agent集群需要框架支撑** | 直接用API调用无法发挥300子代理能力 | 配合OpenClaw、AutoGen、或本系统的Orchestrator |
| **视觉输入按帧计费** | 视频处理成本高 | 先抽关键帧，只传关键帧而非完整视频 |
| **长会话后期一致性下降** | 超长时间运行的Agent可能"忘记"早期设定 | 每50轮对话重新注入system prompt，或拆分任务 |
| **创造性输出偏弱** | 模糊需求下的创新方案质量不稳定 | 需求描述尽量具体化、结构化 |

## 三、Kimi Code 评估与替代方案

### 3.1 Kimi Code 为什么"不好用"

根据社区反馈和实际测试，Kimi Code存在以下具体问题：

| 问题类别 | 具体表现 | 根本原因 |
|----------|----------|----------|
| **补全延迟** | 代码补全有明显延迟，不如Cursor顺滑 | 插件架构的固有延迟 + 网络请求开销 |
| **上下文理解浅** | 只能理解当前文件，跨文件引用不准确 | 缺乏项目级索引，仅基于当前编辑器上下文 |
| **Agent能力弱** | 无法执行多步骤任务（如自动创建多文件） | 作为IDE插件，缺乏文件系统操作权限 |
| **UI交互差** | 对话框体验不如专门的AI IDE | 受制于VS Code的Webview能力 |
| **模型切换不便** | 无法灵活切换不同模型应对不同任务 | 只支持Kimi系列模型 |
| **中文场景优化不足** | 中文注释和文档生成的质量不如英文 | 训练语料中英文比例失衡 |

**结论**: Kimi Code的定位是"VS Code里的AI聊天插件"，不是一个完整的AI编程环境。对于你这种需要多Agent协作、项目级理解的场景，它的能力天花板很明显。

### 3.3 首选推荐：Trae 国内版

**推荐理由（按匹配度排序）**：

1. **中文原生**：全中文界面、中文AI对话质量明显优于其他工具 → 符合你"不看英文"的偏好
2. **完全免费**：目前无功能限制的免费版 → 零成本迁移
3. **支持Kimi模型**：可以直接调用Kimi K2.6作为底层模型 → 保持你的技术栈一致
4. **Builder模式**：用自然语言描述需求，自动拆解成多个文件/步骤实现 → 符合"不看代码细节"的偏好
5. **VS Code兼容**：一键导入VS Code的插件、主题、快捷键 → 迁移无门槛
6. **字节出品**：持续迭代，国内访问稳定 → 不需要科学上网

### 3.4 备选方案：Cursor

如果你未来需要处理更复杂的任务，或者Trae的Builder模式不能满足需求，可以考虑Cursor。

**Cursor的优势场景**：
- 需要同时编辑10+个文件的大型重构
- 需要Claude/GPT-4o等国外模型的能力
- 需要Composer模式进行复杂的多文件协调

**Cursor配置Kimi K2.6的方法**：
```
Settings → AI Models → Add Model
→ 填入OpenAI兼容配置：
  - Base URL: https://api.moonshot.cn/v1
  - API Key: 你的Moonshot API Key
  - Model: kimi-k2.6
```

**Cursor的Composer模式使用技巧**：
```
快捷键 Ctrl+I 打开Composer
→ 输入自然语言需求
→ Cursor会分析需要修改哪些文件
→ 逐文件展示修改（绿色=新增，红色=删除）
→ 点击Apply确认修改
```

### 4.2 各Agent详细配置

#### Agent 1: 代码编写Agent（核心）

| 配置项 | 建议 |
|--------|------|
| **驱动模型** | Kimi K2.6（最强代码能力） |

## 八、常见问题FAQ

**Q: Trae国内版和Kimi Code有什么区别？**
A: Trae是完整的AI原生IDE，有Builder模式可以自动创建多文件项目；Kimi Code只是VS Code的聊天插件，只能做单轮对话和简单补全。相当于"智能手机"和"功能手机"的区别。

**Q: 我已经习惯了VS Code，换Trae麻烦吗？**
A: Trae基于VS Code内核，可以一键导入所有配置（插件、主题、快捷键），迁移成本几乎为零。

**Q: K2.6的Agent Swarm能力怎么用？**
A: K2.6本身支持300个子代理，但需要通过编排框架（如本系统的Orchestrator）来调度。模型层面提供能力，框架层面组织执行。

**Q: 只用Kimi不用其他模型可以吗？**
A: 完全可以。K2.6的代码能力已经达到业界顶级（SWE-Bench Pro 58.6%超过GPT-5.4）。建议的模型分层主要是为了成本控制，K2.5处理简单任务更便宜。

**Q: Windows环境下有什么特别注意的？**
A: Trae国内版对Windows支持良好。注意安装路径不要含中文，Python虚拟环境建议使用venv而非conda（路径问题更少）。

**Q: "不看代码细节"这个偏好怎么落地？**
A: 配置Agent的输出格式——要求只输出"做了什么"和"结论"，代码细节写入文件而非展示。Orchestrator汇总时只提取状态信息。
