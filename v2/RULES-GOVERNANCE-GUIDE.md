## P1: 规则可视化体系

### 1.1 设计原则

| 原则 | 说明 |
|------|------|
| 一眼可见 | 打开文件30秒内知道当前有多少规则、状态如何 |
| 层次清晰 | 元规则/执行规则/场景规则三层分明 |
| 状态明确 | 每条规则处于什么阶段，清清楚楚 |
| 零门槛 | Markdown格式，任何文本编辑器都能打开 |
| 单点入口 | 所有规则信息汇总在一个文件，不分散 |

### 1.2 规则层次模型

```
                    ┌──────────────────┐
                    │    元规则 (M)     │  ← 最多3条，AI绝对不能自行修改
                    │  Meta-Rules      │     定义"规则怎么管"
                    │  [0/3]           │
                    └────────┬─────────┘
                             │ 约束
           ┌─────────────────┼─────────────────┐
           ▼                 ▼                 ▼
    ┌─────────────┐   ┌─────────────┐   ┌─────────────┐
    │  执行规则   │   │  执行规则   │   │  执行规则   │  ← 最多10条
    │ (E1) 编码   │   │ (E2) 测试   │   │ (E3) 文档   │     定义"开发怎么做"
    └─────────────┘   └─────────────┘   └─────────────┘
           │                 │                 │
           ▼                 ▼                 ▼
    ┌─────────────┐   ┌─────────────┐   ┌─────────────┐
    │ 场景规则 S1 │   │ 场景规则 S2 │   │ 场景规则 S3 │  ← 最多10条
    │ 前端Vue     │   │ 后端API     │   │ 数据库      │     定义"特定场景补充"
    └─────────────┘   └─────────────┘   └─────────────┘
```

**三层定义**：

| 层次 | 标识 | 定义 | 上限 | 谁可以改 |
|------|------|------|------|----------|
| 元规则 | M1-M3 | 关于规则本身的规则 | 3条 | 只有用户 |
| 执行规则 | E1-E10 | 每次开发都要遵守的操作标准 | 10条 | 用户批准后AI执行 |
| 场景规则 | S1-S10 | 特定技术栈/模块/场景的补充 | 10条 | 用户批准后AI执行 |

### 1.3 规则状态定义

每条规则在任何时刻都处于以下五种状态之一：

| 状态 | 图标 | 含义 | 谁可以转移 |
|------|------|------|------------|
| 提案中 | 📝 | 已提交变更提案，等待用户审批 | 用户审批后转移 |
| 试运行 | 🧪 | 用户批准试运行，7天观察期 | 7天后自动转正，或用户提前终止 |
| 生效 | ✅ | 正常执行中 | 用户批准废弃后转移 |
| 待废弃 | ⏳ | 用户已批准废弃，等待执行 | AI执行归档后转移 |
| 已归档 | 🗃️ | 已废弃，保留6个月后移除 | 6个月后AI自动移除 |

### 1.4 可视化文件结构

```
RULES-GOVERNANCE.md (规则治理看板)
│
├── 一、规则统计仪表盘      ← 数字总览，一眼看健康度
├── 二、规则层次总览        ← 金字塔 + 三层详细表格
├── 三、规则状态图          ← 生命周期图 + 各状态计数
├── 四、规则分布热力图       ← 规则分散在哪些文件中
├── 五、最近变更记录        ← 最近5次变更摘要
├── 六、我的确认待办        ← 用户需要确认的提案列表
└── 七、AI行为自检声明       ← AI每次回复前必须自检
```

### 1.5 文件位置

| 文件 | 路径 | 作用 |
|------|------|------|
| 规则治理看板 | `/RULES-GOVERNANCE.md` | 唯一入口，所有可视化 |
| 变更日志 | `/RULES-CHANGELOG.md` | 只追加的历史记录 |
| 提案模板 | `/RULES-PROPOSAL-TEMPLATE.md` | 变更提案的标准格式 |

> 建议：三个文件都放在项目根目录，与 AGENTS.md 同级，确保AI每次都能读取到。

## P2: 规则变更确认机制

### 2.1 变更流程总图

```
触发变更的场景:
├── 1. P0/P1事故后 → AI总结经验教训
├── 2. 用户主动提出 → "我们加个规则..."
├── 3. AI检测到模式 → 发现反复出现的问题
└── 4. 规则冲突/过期 → 需要调整

变更流程:

  触发
    │
    ▼
┌─────────────────────────────────────┐
│  Step 1: AI判断是否需要新增规则      │
│  (而不是放入MEMORY.md作为参考)       │
└─────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────┐
│  Step 2: AI填写变更提案              │
│  (使用 RULES-PROPOSAL-TEMPLATE.md)   │
│  包含：变更详情 + 理由 + 影响分析     │
└─────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────┐
│  Step 3: AI在RULES-GOVERNANCE.md中   │
│  "提案中"区域创建待确认条目           │
│  状态 = 📝 提案中                     │
└─────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────┐
│  Step 4: AI向用户展示完整提案        │
│  并明确请求确认                       │
│  "请确认：[同意/否决/需要修改] ?"     │
└─────────────────────────────────────┘
    │
    ▼
         ┌──────────────┬──────────────┬──────────────┐
         ▼              ▼              ▼              ▼
      [批准]        [试运行]       [否决]         [修改]
         │              │              │              │
         ▼              ▼              ▼              ▼
    ┌────────┐    ┌────────┐    ┌────────┐    回到Step 2
    │立即生效 │    │7天观察 │    │不执行  │    修改提案
    │更新状态 │    │7天后转 │    │记录否决│    重新提交
    │✅生效   │    │正为✅  │    │理由   │
    └────────┘    └────────┘    └────────┘
```

### 2.2 确认节点详解

#### 确认位置
- **确认发生在对话中**：AI展示提案 → 用户在对话中回复
- **不在文件中确认**：用户不需要编辑任何文件来"签署"提案
- **AI负责记录**：用户口头（对话中）确认后，AI负责更新所有文件

#### 有效确认语句（AI判断标准）

用户的以下回复视为**有效确认**：
- "同意" / "批准" / "可以" / "没问题"
- "试运行" / "先试运行看看" / "试一周"
- "加上吧" / "采纳" / "执行"
- "同意，但改一下..."（附带修改意见，视为有条件批准）

用户的以下回复视为**否决或暂缓**：
- "不同意" / "否决" / "不用"
- "先不处理" / "以后再说" / "暂缓"
- "这个不需要变成规则" / "放在MEMORY里就行"
- （沉默/不回复）← 绝对不等于同意

#### AI提示语模板

```
【规则变更提案 - 等待您的确认】

我检测到 [触发原因]，建议 [新增/修改/废弃] 规则 [规则名称]。

详细提案如下：
[...提案内容...]

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
请回复您的决定：
• "同意" — 立即实施
• "试运行" — 试运行7天
• "否决" — 不实施，理由：___
• "需要修改" — 我的意见：___

您的回复将决定这条规则是否生效。
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 2.3 变更记录规范

每次确认后的变更必须在 `RULES-CHANGELOG.md` 中记录，格式：

```markdown
### CHG-YYYY-MM-DD-NNN: [变更标题]

| 字段 | 内容 |
|------|------|
| 变更日期 | YYYY-MM-DD |
| 变更类型 | 新增 / 修改 / 废弃 |
| 规则ID | [如 E4] |
| 规则名称 | [名称] |
| 变更前 | [内容或N/A] |
| 变更后 | [内容或N/A] |
| 变更原因 | [理由] |
| 提案人 | AI / 用户 |
| 确认人 | 用户 |
| 确认方式 | "[用户的原话]" |
```

### 2.4 AI修改规则的权限矩阵

| 操作 | 元规则(M) | 执行规则(E) | 场景规则(S) | 治理文件 |
|------|-----------|-------------|-------------|----------|
| **读取** | ✅ 随时 | ✅ 随时 | ✅ 随时 | ✅ 随时 |
| **新增提案** | ❌ 不可 | ✅ 需用户确认 | ✅ 需用户确认 | ✅ 需用户确认 |
| **直接新增** | ❌ 绝对禁止 | ❌ 禁止 | ❌ 禁止 | ❌ 禁止 |
| **修改提案** | ❌ 不可 | ✅ 需用户确认 | ✅ 需用户确认 | ✅ 需用户确认 |
| **直接修改** | ❌ 绝对禁止 | ❌ 禁止 | ❌ 禁止 | ▶️ 仅统计数字 |
| **废弃提案** | ❌ 不可 | ✅ 需用户确认 | ✅ 需用户确认 | ✅ 需用户确认 |
| **自检声明填写** | — | — | — | ✅ AI必填 |

**特别说明**：
- AI可以在 `RULES-GOVERNANCE.md` 中更新**统计数字**（如规则计数、状态计数），但不得修改**规则内容条目**
- AI每次回复前必须填写"AI行为自检声明"区域
- 用户确认后的规则更新，AI必须引用用户的原话作为授权凭证

### 2.5 违规检测机制

#### AI自行添加规则的检测方法

**方法一：规则数量突变检测**

AI每次会话开始时必须：
1. 读取 RULES-GOVERNANCE.md
2. 记录当前规则总数
3. 会话结束时对比总数
4. 如果总数增加但用户没有确认记录 → 触发自我告警

**方法二：关键词扫描**

AI不得在任何文件中写入以下规则性语言，除非有用户确认：
- "必须..." / "一定要..." / "务必..."
- "禁止..." / "不得..." / "不允许..."
- "每次都要..." / "永远..." / "绝对..."
- "如果不...就要..."（条件性强制）

**例外**：当前已生效规则中的规则性语言不受此限制。

**方法三：文件差异自检**

AI在修改任何文件前，必须自问：
```
这个修改是否：
□ 增加了新的"应该/必须/禁止"类语句？
□ 创建了新的小标题包含"规则""规范""标准"？
□ 在没有用户确认的情况下扩大了规则范围？

如果任一勾选为"是" → 停止修改，转为提案模式
```

**用户侧检测**：

如果用户发现AI可能自行添加了规则，可以：
1. 问AI："你现在执行多少条规则？列出来"
2. 对比 RULES-GOVERNANCE.md 中的记录
3. 如果发现未记录的规则 → AI违规

## P3: 规则膨胀自动检测

### 3.1 检测指标体系

| 指标 | 检测时机 | 检测方式 | 阈值 |
|------|----------|----------|------|
| 规则总数 | 每次规则变更时 | AI自动计数 | **20条** |
| 单层次数量 | 每次新增规则时 | AI按层次计数 | 元3 / 执行10 / 场景10 |
| 涉及文件数 | 每次新增规则时 | 统计有规则的文件 | **5个文件** |
| 文件总大小 | 每次规则变更时 | AI估算 | **50KB** |
| 周变更频率 | 每周一 | 统计上周变更次数 | **≤3次/周** |
| 试运行规则数 | 每次变更时 | 统计🧪状态规则 | **≤2条** |

### 3.2 三级告警机制

```
规则数量告警级别：

  0-15条  [🟢 健康]    — 正常运行
 16-18条  [🟡 警告]    — 触发第一次告警
 19-20条  [🟠 紧急]    — 触发紧急告警，禁止新增
   >20条  [🔴 超标]    — 强制进入规则清理流程
```

#### 🟡 黄色警告 (数量≥16条)

触发动作：
- AI自动在看板中标注 🟡
- AI向用户发出警告：

```
【规则膨胀警告 🟡】

当前规则数量: X/20，已接近上限。

建议：
1. 审视是否有可以合并的规则
2. 审视是否有可以废弃的过时规则
3. 考虑将部分执行规则降级为场景规则

是否需要我提供规则合并建议？
```

#### 🟠 橙色警告 (数量≥19条)

触发动作：
- AI自动在看板中标注 🟠
- **禁止新增规则**（所有新增提案自动挂起）
- AI向用户发出紧急告警：

```
【规则膨胀紧急警告 🟠】

当前规则数量: X/20，即将达到上限！

⚠️ 在规则数量降至16条以下之前，我将暂停所有新增规则提案。

强烈建议立即进行规则清理：
□ 审视执行规则，将不常用的降级为场景规则
□ 审视场景规则，废弃过时的
□ 合并重复或相似的规则

是否需要我提供规则清理方案？
```

#### 🔴 红色超标 (数量>20条)

触发动作：
- AI自动在看板中标注 🔴
- **冻结所有规则新增**
- **强制启动规则清理流程**

```
【规则膨胀超标 🔴】

当前规则数量: X/20，已超出上限！

❌ 所有规则新增已冻结
❌ 所有规则修改已冻结（除清理外）

必须执行规则清理流程后才能恢复。
我正在准备规则清理方案，请稍候...
```

### 3.3 自动清理流程（超标后）

当规则数量 > 20条时，AI自动执行：

```
Step 1: AI扫描所有规则，标记候选清理项：
        - 超过30天未触发的场景规则 → 建议废弃
        - 与其他规则重复度>80%的规则 → 建议合并
        - 标记为"🧪 试运行"超过14天的规则 → 建议转正或废弃
        - 用户曾经表示"不太重要"的规则 → 建议废弃

Step 2: AI生成清理提案，提交用户确认
        （格式同规则变更提案）

Step 3: 用户确认后执行清理

Step 4: 数量降至 ≤16条 后解除冻结
```

### 3.4 膨胀防护墙（预防性措施）

除了超标后的处理，还有三道预防性防线：

**第一道：新增门槛**
- 新增规则前，AI必须论证"为什么这条必须变成规则，而不是放在MEMORY.md"
- 如果可以用检查清单/模板/示例代替 → 不升级为规则

**第二道：合并推荐**
- 当新提案与现有规则相似度 > 60% → AI建议合并而非新增
- 相似度检测由AI自行判断

**第三道：定期审视**
- 每满10次变更 → AI主动提议审视现有规则
- 每月1日 → AI自动列出"30天未触发"的场景规则，询问是否保留

## AI行为约束清单

### 必须做的事（Must）

| # | 行为 | 时机 | 未执行的后果 |
|---|------|------|-------------|
| M1 | 读取 RULES-GOVERNANCE.md | 每次会话开始时 | 可能违反未知规则 |
| M2 | 填写AI行为自检声明 | 每次回复前 | 无法自证合规 |
| M3 | 使用提案模板申请变更 | 任何规则变更前 | 视为违规操作 |
| M4 | 获得用户明确确认 | 执行变更前 | 变更无效，需回滚 |
| M5 | 更新变更日志 | 变更执行后 | 历史记录缺失 |
| M6 | 检查规则数量是否超标 | 每次变更前后 | 可能触发膨胀 |
| M7 | 统计数字与实际一致 | 更新看板时 | 数据失真 |

### 绝对禁止的事（Must NOT）

| # | 禁止行为 | 违规示例 | 发现后的处理 |
|---|----------|----------|-------------|
| N1 | 未经确认新增规则 | 事故后自动写"以后必须..." | 立即废弃，回滚文件 |
| N2 | 未经确认修改元规则 | 将"执行前确认"改为"执行后通知" | 立即回滚，告警用户 |
| N3 | 经验教训自动升级为规则 | MEMORY.md中的经验变成AGENTS.md中的强制 | 降级回MEMORY.md |
| N4 | 拆分规则规避上限 | 将1条规则拆成3条子规则 | 合并回原状 |
| N5 | 在普通文档中植入规则 | 在README中写"每次提交必须..." | 移除规则性语言 |
| N6 | 以"紧急"为由绕过确认 | "这个很紧急，我先加上您后面再看" | 停止执行，走正常流程 |
| N7 | 沉默=同意的假设 | 用户没回复就当同意了 | 确认无效，必须重新申请 |
| N8 | 删除或修改CHANGELOG记录 | 把旧记录删掉让文件变短 | 不可删除，只能追加 |

### 违规处理流程

```
发现AI违规（用户发现或AI自检发现）
    │
    ▼
标记违规变更
    │
    ▼
┌──────────────────────────┐
│ 1. 回滚违规修改           │
│    恢复文件到变更前状态    │
└──────────────────────────┘
    │
    ▼
┌──────────────────────────┐
│ 2. 在看板中记录违规事件    │
│    日期 + 违规类型 + 涉及规则 │
└──────────────────────────┘
    │
    ▼
┌──────────────────────────┐
│ 3. AI向用户道歉并说明     │
```

## 常见问题

### Q1: 这个方案会不会降低开发效率？

不会。原因：
- 日常开发不需要看这些文件，AI自动读取
- 只有规则变更时才需要用户确认，不是每次对话
- 快速模式提案只需一句话确认
- 规则清晰后，AI执行更准确，反而减少返工

### Q2: AI会乖乖遵守这些约束吗？

不能完全保证，但有四道防线：
1. **自检声明**：AI每次回复前自查
2. **数量检测**：规则增加必有痕迹
3. **变更日志**：所有变更有记录可追溯
4. **用户检测**：用户随时可以让AI列出所有规则核对

### Q3: 规则上限20条够不够用？

够用。设计思路：
- 元规则3条 = 规则本身的规则（极少变动）
- 执行规则10条 = 真正的核心行为准则
- 场景规则10条 = 按需启用，不累计生效

如果20条不够，说明规则粒度过细，应该合并。

### Q4: 用户怎么快速检查AI是否违规？

三种方法，从简到繁：

**快速检查（10秒）**：
```
用户："当前有多少条规则？列出来"
AI："当前X条规则：[列表]"
用户对比看板中的数字
```

**标准检查（1分钟）**：
```
用户："打开 RULES-GOVERNANCE.md，读给我听"
AI：读取文件内容
用户核对规则列表
```

**深度检查（5分钟）**：
```
用户："扫描所有文件，找出所有包含'必须/禁止/每次'的句子"
AI：扫描并列出
用户对比看板，找出未记录的
```
