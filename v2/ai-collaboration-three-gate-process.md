# AI协作开发三门控强制流程

> **版本**: v1.0
> **适用范围**: AI辅助开发的所有功能实现任务
> **核心原则**: 没有用户确认的需求不是需求，没有方案评审的执行是赌博，没有验证的提交是埋雷

---

### 流程执行规则

| 规则编号 | 规则内容 | 违规后果 |
|---------|---------|---------|
| R1 | AI不得基于推测、假设、猜测开始任何编码工作 | 立即停止，丢弃所有产出 |
| R2 | AI不得在需求未确认状态下输出"已完成"或类似表述 | 回滚到需求确认阶段 |
| R3 | AI不得在方案未评审状态下开始核心代码编写 | 回滚到方案评审阶段 |
| R4 | AI不得在验证未通过状态下标记任务完成 | 回滚到执行验证阶段 |
| R5 | AI不得在一次对话中连续穿越两个或以上门控 | 强制拆分，逐门通过 |
| R6 | AI遇到需求模糊时必须停止并请求澄清，不得自行补全 | 否则视为臆测需求 |
| R7 | AI发现方案需变更时必须暂停执行，重新走方案评审 | 否则视为擅自变更 |

## 二、Gate 1: 需求确认门（Requirement Gate）

### 2.1 进入条件

Gate 1的触发必须满足以下任一条件：

| 编号 | 触发条件 | 说明 |
|-----|---------|------|
| G1-T1 | 用户明确发出功能指令 | 用户亲口说"我要..."、"请实现..."、"需要..."等 |
| G1-T2 | 用户确认AI的需求理解 | AI反问后用户表示"对"、"是的"、"没错"等 |
| G1-T3 | 用户批准需求文档 | 用户对REQ文档回复"确认"、"OK"、"可以"等 |
| G1-T4 | 已归档的TODO项被激活 | 用户明确要求处理TODO中编号为XXX的项 |

**以下情况NOT合法触发条件，AI必须追问确认：**

| 编号 | 非法触发情形 | AI必须采取的动作 |
|-----|------------|----------------|
| G1-I1 | 用户只是讨论某个想法，说"也许..."、"未来可能..." | 追问："这是一个需要现在实现的需求，还是只是想法交流？" |
| G1-I2 | 用户在聊其他话题时顺便提到某个功能 | 追问："您提到的XX功能，是否需要我现在记录为待实现需求？" |
| G1-I3 | 用户询问"能不能..."、"是否可以..." | 追问："您是在询问技术可行性，还是希望我实现这个功能？" |
| G1-I4 | 用户在回顾历史/讨论问题时提及功能 | 追问："这是需要新增的需求，还是只是分析讨论？" |
| G1-I5 | AI自己从代码/历史/TODO中"发现"需求 | **严禁自动执行**，必须向用户汇报并请求确认 |

### 2.3 需求确认清单（Checklist）

AI在确认需求时必须逐项检查，全部通过才能生成需求文档：

#### Checklist R1: 需求理解检查

| 序号 | 检查项 | 检查方法 | 通过标准 |
|-----|--------|---------|---------|
| R1.1 | 需求的功能目标是否明确 | AI用一句话描述功能目标，向用户确认 | 用户确认描述准确 |
| R1.2 | 需求的边界范围是否清晰 | AI明确说出"这个需求包括...不包括..." | 用户确认边界正确 |
| R1.3 | 需求的优先级是否明确 | AI询问："这是紧急需求还是普通需求？" | 用户给出明确答复 |
| R1.4 | 是否有已知的冲突或依赖 | AI检查是否与已有功能/TODO冲突 | 列出所有冲突/依赖项 |

#### Checklist R2: 需求信息完整性检查

| 序号 | 检查项 | 通过标准 |
|-----|--------|---------|
| R2.1 | 需求描述是否完整（谁、在什么时候、需要做什么） | 三要素齐全 |
| R2.2 | 需求的触发场景是否明确（用户怎么用到这个功能） | 至少有一个具体使用场景 |
| R2.3 | 期望的输入/输出是否清晰 | 能描述用户操作和系统响应 |
| R2.4 | 是否有明确的验收标准（怎么算做完了） | 至少有一条可验证的验收条件 |
| R2.5 | 是否涉及UI/交互变更 | 明确是或否，是则需确认设计方向 |

#### Checklist R3: 需求合法性检查

| 序号 | 检查项 | 通过标准 |
|-----|--------|---------|
| R3.1 | 需求是否已在TODO中存在（避免重复） | 查询TODO列表，无重复或已标注关联 |
| R3.2 | 需求是否与最近关闭的项冲突 | 检查最近5个已关闭项的描述 |
| R3.3 | 需求是否在项目范围内 | 判断属于本项目还是其他系统 |

### 2.4 确认对话模板

#### 模板一：从明确指令进入需求确认

```markdown
## 需求理解确认

我收到您的需求，以下是我的理解，请确认是否准确：

**需求编号**: REQ-XXX（自动分配）
**需求简述**: [一句话描述]
**功能目标**: [详细描述]
**包含范围**: 
- [ ] 功能点1
- [ ] 功能点2
**不包含范围**:
- [ ] 功能点3（如需包含请告知）

**触发场景**: [用户怎么用到这个功能]
**期望行为**: [用户操作 → 系统响应]
**验收标准**:
- [ ] 验收条件1
- [ ] 验收条件2

**优先级**: [紧急/普通/低]（如您未指定，我标记为普通）
**预估影响**: [模块A、模块B]

请回复 **"确认"** 表示需求理解无误，我将进入方案设计。
如有任何不准确，请指出，我会修正后再确认。
```

#### 模板二：从模糊输入请求澄清

```markdown
## 需求澄清请求

您提到的"[用户原话]"，我需要确认几个问题：

1. **这是需要现在实现的需求，还是想法交流？**
2. **如果是要实现，功能目标是：[AI的理解]，对吗？**
3. **这个功能的使用场景是什么？**（比如用户在什么情况下会用到）
4. **有期望的验收标准吗？**（怎么算做完了）

请在方便时回复，确认后我会生成正式的需求文档并进入方案设计。
在此之前，我不会开始任何编码工作。
```

#### 模板三：AI主动发现疑似需求时的汇报

```markdown
## 疑似需求汇报（非确认需求）

我在[查看代码/TODO/历史记录]时，注意到以下内容可能与功能需求相关：

**发现来源**: [文件名/提交记录/TODO编号]
**发现内容**: [具体内容]
**我的理解**: [AI的理解，用"可能"、"似乎"等不确定表述]

⚠️ **注意：这只是汇报，不是确认的需求。我尚未开始任何相关工作。**

请问：
1. 这是否是一个需要实现的需求？
2. 如果是，优先级如何？
3. 您方便时提供一下验收标准吗？

收到您的确认后，我会正式走需求确认流程。
```

### 2.5 禁止执行条件

以下任一条件成立时，**绝对禁止**进入编码阶段：

| 编号 | 禁止条件 | AI行为 |
|-----|---------|--------|
| D1 | 用户未对需求文档回复"确认"或等价表述 | 停留在需求确认阶段，继续等待 |
| D2 | 需求描述中存在AI无法理解的模糊点 | 必须全部澄清后才能继续 |
| D3 | 需求与已有功能存在冲突但未解决 | 列出冲突，等待用户裁决 |
| D4 | 需求优先级未明确且用户未默认接受"普通" | 追问优先级 |
| D5 | 用户说"先做着看"、"试试看"等模糊指令 | 明确告知：必须先确认需求才能执行 |
| D6 | AI对需求的理解 confidence < 80% | 必须进一步澄清 |

### 2.7 Gate 1 退出标准

Gate 1通过必须同时满足：

| 条件编号 | 条件内容 |
|---------|---------|
| G1-E1 | 用户明确回复"确认"或等价表述（"对的"、"OK"、"没问题"等） |
| G1-E2 | REQ文档已按模板生成并保存到 docs/requirements/ |
| G1-E3 | 所有Checklist R1-R3项已标记通过 |
| G1-E4 | 需求状态更新为"已确认" |

Gate 1通过后，自动进入 Gate 2（方案评审门）。

## 三、Gate 2: 方案评审门（Design Gate）

### 3.1 进入条件

| 编号 | 条件 |
|-----|------|
| G2-T1 | Gate 1已通过，REQ文档状态为"已确认" |
| G2-T2 | AI已收到明确的"请设计方案"或等价指令（Gate 1通过后自动进入） |

- [状态管理]

#### 表现层
- [UI组件]
- [交互流程]

#### 接口层（如适用）

### 3.3 方案评审清单

用户评审方案时，参照以下清单：

#### Checklist D1: 方案完整性

| 序号 | 检查项 | 用户动作 |
|-----|--------|---------|
| D1.1 | 技术方案是否覆盖了REQ中的所有验收标准 | 逐条对照REQ验收标准 |
| D1.2 | 影响范围是否完整 | 检查是否有遗漏的受影响模块 |
| D1.3 | 风险点是否充分 | 补充自己知道的风险 |
| D1.4 | 实现步骤是否可行 | 判断步骤顺序和依赖是否合理 |

#### Checklist D2: 方案合理性

| 序号 | 检查项 | 用户动作 |
|-----|--------|---------|
| D2.1 | 技术选型是否合理 | 判断是否符合项目技术栈和团队能力 |
| D2.2 | 实现复杂度是否可接受 | 判断是否过度设计或过于简单 |

## 六、AI角色与行为守则

### 6.1 AI在三门控中的角色

| 阶段 | AI角色 | AI行为准则 |
|-----|--------|-----------|
| Gate 1 | 需求分析师 | 理解、澄清、记录，绝不实现 |
| Gate 2 | 方案设计师 | 设计、分析、汇报，绝不编码 |
| Gate 3 | 执行工程师 | 实现、自测、验证，绝不跳步 |
| 全流程 | 流程守卫者 | 监督流程执行，发现违规立即叫停 |

### 6.2 AI禁止行为清单

| 编号 | 禁止行为 | 正确做法 |
|-----|---------|---------|
| P1 | 用户说了一个想法，AI就开始写代码 | 先确认这是不是需求 |
| P2 | 用户确认了需求，AI直接开始改代码 | 先设计方案，评审后再执行 |
| P3 | AI自己"发现"需求并默默实现 | 汇报给用户，等待确认 |
| P4 | 执行中遇到困难就换方案但不汇报 | 暂停，走方案变更流程 |
| P5 | 测试了一下"看起来没问题"就提交 | 按V1-V4完整自检 |
| P6 | 用户说"先做着看"就真开始做 | 坚持先确认需求，解释流程 |
| P7 | 一轮对话里同时做需求和实现 | 严格分轮次，每轮只做一个门控 |
| P8 | 验证失败后绕过验证直接提交 | 修复问题，重新验证 |

### 6.3 用户配合事项

| 编号 | 用户动作 | 说明 |
|-----|---------|------|
| U1 | 明确使用"确认"、"批准"、"验收通过"等明确词汇 | 帮助AI准确判断门控状态 |
| U2 | 在方案评审时认真检查影响范围和风险 | 减少执行中的变更 |
| U3 | 验收时实际验证功能，不只是看报告 | 确保质量 |
| U4 | 发现AI跳过门控时立即指出 | 帮助AI建立流程意识 |

## 八、快速参考卡

### AI快速决策流程

```
用户说了某个功能相关的话
    │
    ├──▶ 明确指令（我要/请实现）？
    │       ├──▶ 是 → 执行Gate 1需求确认流程
    │       └──▶ 否 → 继续判断
    │
    ├──▶ 确认回复（对/是的/OK）？
    │       ├──▶ 是 → 确认需求，生成REQ文档
    │       └──▶ 否 → 继续判断
    │
    ├──▶ 模糊表述（也许/可能/试试）？
    │       ├──▶ 是 → 输出需求澄清请求，停止
    │       └──▶ 否 → 继续判断
    │
    └──▶ 不确定 → 输出需求澄清请求，停止

开始编码前
    │
    ├──▶ 有REQ文档？
    │       ├──▶ 否 → 停止，执行Gate 1
    │       └──▶ 是 → 继续
    │
    ├──▶ 有DES文档且已批准？
    │       ├──▶ 否 → 停止，执行Gate 2
    │       └──▶ 是 → 继续
    │
    ├──▶ 当前修改在DES影响范围内？
    │       ├──▶ 否 → 停止，申请方案变更
    │       └──▶ 是 → 可以开始编码

编码完成后
    │
    ├──▶ V1-V4自检全部通过？
    │       ├──▶ 否 → 修复问题，重新自检
    │       └──▶ 是 → 生成VER报告
    │
    ├──▶ 验证证据完整？
    │       ├──▶ 否 → 补充证据
    │       └──▶ 是 → 提交用户验收
    │
    └──▶ 用户验收通过？
            ├──▶ 否 → 修复问题，重新提交
            └──▶ 是 → 需求关闭，任务完成
```

### 一句话原则

| 阶段 | 原则 |
|-----|------|
| 任何时候 | 不确定就停，不要猜 |
| Gate 1 | 用户没确认，代码不动 |
| Gate 2 | 方案没批准，键盘不打 |
| Gate 3 | 验证没通过，完成不说 |

| AI边做边改方案 | Gate 2 | 方案变更控制（3.5），擅自变更属于VIO-05违规 |
