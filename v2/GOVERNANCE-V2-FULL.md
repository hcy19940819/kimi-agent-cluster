### 核心设计: 三层金字塔 + 可视化看板

```
                    ┌──────────────┐
                    │   元规则 M    │  ← 最多3条，只有你能改
                    │  [0/3]       │
                    └──────┬───────┘
                           │
           ┌───────────────┼───────────────┐
           ▼               ▼               ▼
    ┌────────────┐  ┌────────────┐  ┌────────────┐
    │  执行规则 E  │  │  执行规则 E  │  │  执行规则 E  │  ← 最多10条
    │  [0/10]     │  │  [编码/测试/ │  │  [文档/提交/ │     需你批准AI才能改
    │             │  │   安全/...]  │  │   验证/...]  │
    └────────────┘  └────────────┘  └────────────┘
                           │
           ┌───────────────┼───────────────┐
           ▼               ▼               ▼
    ┌────────────┐  ┌────────────┐  ┌────────────┐
    │  场景规则 S  │  │  场景规则 S  │  │  场景规则 S  │  ← 最多10条
    │  [前端Vue]   │  │  [后端API]   │  │  [数据库]    │     需你批准AI才能改
    │  [人脸模块]  │  │  [去重模块]  │  │  [...]       │
    └────────────┘  └────────────┘  └────────────┘

规则总数: X/20（上限20条）
状态: 🟢健康 🟡警告 🟠紧急 🔴超标
```

### 配套文件（放在项目根目录）

| 文件 | 作用 | 谁编辑 |
|------|------|--------|
| `RULES-GOVERNANCE.md` | **可视化看板**：规则列表、状态、统计 | AI维护，你只确认 |
| `RULES-CHANGELOG.md` | 变更历史（只追加，不删除） | AI追加，你不编辑 |
| `RULES-PROPOSAL-TEMPLATE.md` | 变更提案模板（AI填表） | AI填写，你审批 |

### 5条元规则（永远不变）

```
MR1. 单一来源 — 一个规则只存在于一个地方，发现重复立即删除最旧的
MR2. 场景驱动 — 规则按使用场景组织，不按文件类型
MR3. 自动化优先 — 能脚本化的规则绝不靠人记
MR4. 约束有界 — 核心规则≤15条，新增必须淘汰旧规则
MR5. 经验≠规则 — 经验教训不能自动升级为规则，升级需审批
```

### 规则变更确认流程（AI必须遵守）

```
触发变更（事故/你要求/AI检测）
    │
    ▼
AI判断：是真需新规则？还是放MEMORY.md参考即可？
    │
    ▼
AI填写 RULES-PROPOSAL-TEMPLATE.md（5分钟）
    │
    ▼
AI在看板中标注"📝 提案中"
    │
    ▼
AI向你展示提案 + 请求确认
    │
    ▼
你回复：同意 / 试运行 / 否决 / 修改
    │
    ▼
确认后AI执行 + 追加CHANGELOG
```

### AI变更权限矩阵

| 操作 | 元规则(M) | 执行规则(E) | 场景规则(S) |
|------|-----------|-------------|-------------|
| 读取 | ✅ | ✅ | ✅ |
| 新增提案 | ❌ 绝对禁止 | ✅ 需你确认 | ✅ 需你确认 |
| 直接新增 | ❌ | ❌ 禁止 | ❌ 禁止 |
| 修改提案 | ❌ | ✅ 需你确认 | ✅ 需你确认 |
| 直接修改 | ❌ | ❌ 禁止 | ❌ 禁止 |
| 废弃提案 | ❌ | ✅ 需你确认 | ✅ 需你确认 |

### 三级告警（自动检测）

| 级别 | 规则数 | 状态 | 动作 |
|------|--------|------|------|
| 🟢 健康 | 0-15条 | 正常 | 无 |
| 🟡 警告 | 16-18条 | 接近上限 | AI建议合并，提醒审视 |
| 🟠 紧急 | 19-20条 | 即将满 | **冻结新增**，必须清理后才能新增 |
| 🔴 超标 | >20条 | 超标 | **强制清理流程**，所有变更暂停 |

### 关键保障: AI六条红线

1. **禁止先斩后奏** — 不得在确认前执行变更
2. **禁止默认同意** — 你不回复 ≠ 同意
3. **禁止拆分规避** — 不得将1条拆成3条
4. **禁止紧急绕过** — 不得以"紧急"为由跳过确认
5. **禁止间接植入** — 不得在README等文件中植入规则
6. **禁止强制升级** — 经验→规则必须经过审批

## 第三部分: 需求→方案→执行 三门控流程

### 总览图

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│  用户输入 → Gate1 需求确认 → Gate2 方案评审 → Gate3 执行验证 → 关闭  │
│              │                 │                 │       │
│              ▼                 ▼                 ▼       │
│           REQ文档           DES文档           VER文档    │
│           (已确认)          (已批准)          (已验收)    │
│                                                         │
│   ═══════════════════════════════════════════════════   │
│   红色禁区: AI不得自行跳过任何门控，否则视为违规执行        │
│   ═══════════════════════════════════════════════════   │
└─────────────────────────────────────────────────────────┘
```

### Gate 1: 需求确认门

**AI的角色**: 需求分析师（只理解记录，绝不编码）

**合法触发条件**（满足任一才能开始）:
- 你明确说"我要..."、"请实现..."
- 你确认AI的理解（"对"、"是的"）
- 你明确要求处理TODO中某编号

**非法触发**（AI必须追问）:
- 你说"也许..."、"未来可能..." → AI追问"是现在实现还是想法交流？"
- 你只是讨论问题提到功能 → AI追问"需要记录为待实现需求吗？"
- 你问"能不能..." → AI追问"是询问可行性，还是希望我实现？"
- **AI自己从代码/历史中发现的需求** → **严禁自动执行，必须汇报请求确认**

**确认Checklist**（AI逐项检查）:

| 检查项 | 通过标准 |
|--------|---------|
| 功能目标是否明确 | 用一句话描述，你确认准确 |
| 边界范围是否清晰 | AI说"包括...不包括..."，你确认 |
| 触发场景是否明确 | 至少一个具体使用场景 |
| 验收标准是否清晰 | 至少一条可验证的"怎么算做完了" |
| 是否与已有需求冲突 | 无冲突或已标注关联 |

**确认对话模板**（AI必须用这个格式）:

```
## 需求理解确认（REQ-XXX）

我收到您的需求，请确认以下理解是否准确：

**需求简述**: [一句话]
**功能目标**: [详细描述]
**包含范围**: [功能点1]、[功能点2]
**不包含范围**: [功能点3]（如需请告知）
**触发场景**: [用户怎么用]
**验收标准**: [怎么算做完]
**优先级**: [紧急/普通/低]

请回复"确认"进入方案设计，或指出需要修正的地方。
```

**Gate 1 通过标准**（全部满足）:
- ✅ 你明确回复"确认"或等价表述
- ✅ REQ文档已保存到 docs/requirements/
- ✅ Checklist全部通过

### Gate 2: 方案评审门

**AI的角色**: 方案设计师（只设计方案，绝不编码）

**方案输出要求**（AI必须输出5部分）:

1. **技术方案** — 实现思路、关键决策、架构影响
2. **影响范围** — 修改哪些模块/文件/配置/数据/接口
3. **风险评估** — 风险描述 + 发生概率 + 应对方案
4. **实现步骤** — 每步做什么 + 改哪些文件 + 回滚方案
5. **验证计划** — 功能验证 + 回归验证 + 边界验证

**评审Checklist**（你评审时对照）:

| 检查项 | 你检查什么 |
|--------|-----------|
| 是否覆盖所有验收标准 | 逐条对照REQ |
| 影响范围是否完整 | 是否有遗漏的受影响模块 |
| 风险是否充分 | 补充你知道的风险 |
| 步骤是否可行 | 顺序和依赖是否合理 |
| 回滚方案是否可靠 | 每步都能安全回退 |
| 验证计划是否可执行 | 能否在实际环境验证 |

**方案汇报模板**（AI必须用这个格式）:

```
## 方案汇报: DES-XXX

基于已确认的需求 REQ-XXX，方案如下：

### 核心思路
[2-3句话]

### 影响范围
- 修改模块: [X个]
- 修改文件: [Y个]
- 主要风险: [Top 2]

### 需要您决策的问题
- Q1: 方案A（简单快速）vs 方案B（扩展性好），您倾向？
- Q2: 是否接受引入[xxx依赖]？

请回复"批准"开始执行，或回复修改意见。
⚠️ 在批准前，我不会修改任何代码。
```

**方案变更控制**（执行中发现需要改方案时）:

```
执行中发现需要变更
    │
    ▼
1. 立即停止编码
    │
    ▼
2. 评估变更影响
    │
    ▼
3. 向你汇报变更申请
    │
    ▼
你批准 → 更新方案 → 重新评审
你拒绝 → 按原方案继续 或 找workaround
```

**Gate 2 通过标准**（全部满足）:
- ✅ 你明确回复"批准"或等价表述
- ✅ DES文档已保存到 docs/designs/
- ✅ 所有需决策的问题已有明确答案

### Gate 3: 执行验证门

**AI的角色**: 执行工程师（按方案实现，完成后自检）

**自检清单**（AI提交前必须逐项验证）:

| 类别 | 检查项 | 通过标准 |
|------|--------|---------|
| **V1 功能** | 验收标准已实现 | 逐条对照REQ |
| **V1 功能** | 正常流程正确 | 手动/单元测试通过 |
| **V1 功能** | 异常流程正确 | 错误处理合理不崩溃 |
| **V2 回归** | 原有功能正常 | 受影响模块测试通过 |
| **V2 回归** | 无新构建错误 | 构建成功 |
| **V3 质量** | 符合编码规范 | ruff/flake8通过 |
| **V3 质量** | 无调试代码残留 | 无console.log/debugger |
| **V4 文档** | 相关文档已更新 | 功能变更已同步 |

**验证证据**（AI必须提供）:
- 测试执行记录（用例+结果）
- 构建结果（成功截图）
- 变更摘要（改了哪些文件）
- 自检清单（逐项通过/未通过）

**验收对话模板**（AI必须用这个格式）:

```
## 验证报告: VER-XXX

REQ-XXX 实现完成，验证结果：

| 检查项 | 状态 |
|--------|------|
| V1 功能验证 | ✅ 通过 |
| V2 回归验证 | ✅ 通过 |
| V3 代码质量 | ✅ 通过 |
| V4 文档验证 | ✅ 通过 |

修改文件: [N]个（新增[N]个，修改[N]个）

请回复"验收通过"关闭需求，或指出需要修正的问题。
⚠️ 在验收通过前，需求状态保持"待验收"。
```

**Gate 3 通过标准**（全部满足）:
- ✅ 你明确回复"验收通过"或等价表述
- ✅ VER文档已保存到 docs/verifications/

## 第四部分: Kimi2.6 工具链优化

### 核心结论

| 项目 | 建议 | 理由 |
|------|------|------|
| **编程工具** | Trae国内版 替代 Kimi Code | 免费、中文原生、支持Kimi K2.6、Builder模式强 |
| **核心模型** | Kimi K2.6 | 256K上下文、300子代理、代码能力顶级 |
| **辅助模型** | Kimi K2.5 | 测试/文档生成用K2.5更省成本 |
| **Agent框架** | 本系统Orchestrator | 已在用，继续优化 |

### Trae国内版迁移步骤（10分钟）

```
1. 下载: https://www.trae.cn → Windows版
2. 安装: 按向导（路径不含中文）
3. 导入: 打开Trae → 设置 → 导入VS Code配置 → 一键导入
4. 配置模型: 右上角AI模型选择 → 添加自定义 → Kimi → 填入API Key → 选kimi-k2.6
5. 试用: 点击Builder → 描述需求 → 看自动创建文件
```

### Kimi K2.6 最佳实践

**Prompt写法四段式**（效果最好）:

```markdown
## 任务
[一句话描述要做什么]

## 上下文
[相关文件/项目背景/已有代码]

## 要求
1. [具体要求1]
2. [具体要求2]
3. [要求3]

## 输出格式
[要什么格式输出，不要什么]
```

**长上下文利用**（256K tokens）:
- 第一轮：传项目地图（文件摘要）建立全局上下文
- 第二轮：补充相关模块
- 第三轮：基于已积累的上下文下达精确任务
- 前缀缓存让多轮对话成本极低

**多模型分层**（降本增效）:

| 任务 | 模型 | 原因 |
|------|------|------|
| 核心代码编写 | K2.6 | 最强代码能力 |
| 复杂架构设计 | K2.6 | 深度推理 |
| 代码审查 | K2.6 | 发现隐蔽问题 |
| 单元测试生成 | K2.5 | 模式化，够用 |
| 文档生成 | K2.5 | 精度要求较低 |
| 简单补全 | K2 | 成本最低 |

### Agent架构建议

```
Orchestrator（K2.6）统一调度
    ├── 代码编写Agent → K2.6 + Trae Builder
    ├── 代码审查Agent → K2.6 + Flake8/Bandit
    ├── 测试Agent     → K2.5 + pytest
    └── 文档Agent     → K2.5 + MkDocs
```

**各Agent的System Prompt模板**见 `kimi26-toolchain-guide.md`
