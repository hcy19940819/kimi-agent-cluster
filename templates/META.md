# META — 元规则与执行约束

> **这是唯一可信来源。所有规则在此定义，不在此处的规则不存在。**
> 
> 修改规则：必须通过"规则生命周期审批"（见 MR5）

---

## Layer 1: 元规则（永远不变）

```
MR1. 单一来源
     一个规则只存在于一个地方。发现重复立即删除最旧的那份。

MR2. 场景驱动  
     规则按使用场景（启动/开发/提交/事故）组织，不按文件类型组织。

MR3. 自动化优先
     能脚本化的规则绝不靠人记。脚本未覆盖的规则视为无效。

MR4. 约束有界
     核心执行规则 ≤ 15条。新增必须淘汰旧规则。

MR5. 经验 ≠ 规则
     经验教训只能在 incident.md 中记录，不能自动升级为规则。
     升级需经过：提案 → 三问检查 → 试运行2周 → 审批。
```

---

## Layer 2: 执行规则（当前 15/15）

### 【自动化层 — 脚本覆盖，无需记忆】

```
R1. 会话启动 → 运行 pre-flight.sh（3步自检）
R2. 代码提交 → 运行 verify.sh（检查清单通过）
R3. 死循环检测 → 同一操作>3次自动终止并报告
```

### 【决策约束层 — 3条红线，违反=违规】

```
R4. 不改需求外内容 — 不自主改UI/加功能/重构，除非明确要求
R5. 执行前确认 — 方案变更/文件迁移/代码写入先汇报，同意再执行
R6. 文档与代码同步 — commit前文档必须先同步
```

### 【质量约束层 — 防止已知问题】

```
R7. 同一Shell命令最多重试3次
R8. 确认对话最多2轮，未决立即上报
R9. 遗留问题必须标记 TODO/KNOWN-ISSUE，禁止静默跳过
R10. 需求必须标明来源 [user/tech-debt/incident#N/assumed]
```

### 【信息管理层 — 保持上下文一致】

```
R11. 一个规则只存在于一个地方（本文件是唯一规则来源）
R12. 当前任务上下文必须写在 CONTEXT.md #Session 中
R13. 事故记录必须写入 incident.md，禁止另建新文件
```

### 【元规则约束层 — 规则自我管理】

```
R14. 新增规则必须淘汰旧规则（总数≤15）
R15. 经验教训≠规则，升级需审批（见 MR5）
```
