# PLAYBOOK — 开发工作流

> 覆盖场景：接收需求 → 编码 → 自测
> 核心约束：3条红线（R4/R5/R6）+ 4条质量规则（R7-R10）

---

## 需求阶段

```
□ 需求是否有明确来源？（R10）
  是 → 标记来源 [user/tech-debt/incident#N]
  否 → 标记 [assumed]，并告知用户"此为推测需求，请确认"

□ 需求是否在 CONTEXT.md #Session 范围内？
  是 → 继续
  否 → 遵循 R5：先汇报 "该需求超出当前会话目标，是否切换？"
```

## 编码阶段

```
□ 是否只修改需求相关内容？（R4）
  检查：每行变更都能追溯到需求
  反例："顺便重构了一下" / "我觉得这个UI可以优化" → 禁止

□ 方案变更是否先汇报？（R5）
  触发条件：实现方式与最初确认的不同
  操作：暂停 → 说明变更原因 → 等待确认 → 继续

□ 同一命令是否已执行>2次？（R7）
  是 → 第3次执行前必须报告"同一命令已执行N次"
  达到3次 → R3触发，自动终止
```

## 自测阶段

```
□ 测试是否通过？
  命令: ./scripts/verify.sh

□ 是否有遗留问题？（R9）
  无 → 提交
  有 → 标记 TODO/KNOWN-ISSUE + 创建跟踪项
```

---

## 禁止事项（红色清单）

```
✗ 不改需求外内容（R4）
✗ 不自主决策方案变更（R5）
✗ 不静默跳过遗留问题（R9）
✗ 同一命令不超过3次（R7）
✗ 确认对话不超过2轮（R8）
```

## 需求来源标记规范（R10）

| 来源 | 标记 | 合法性 |
|------|------|--------|
| 用户明确需求 | `[user]` | 合法 |
| 技术债务 | `[tech-debt]` | 合法，需说明 |
| 事故修复 | `[incident#N]` | 合法 |
| 推测/假设 | `[assumed]` | 需用户确认，不可标记 completed |
