# MEMORY.md 15KB → 新架构压缩方案

> 本文档详细展示如何将15KB且持续增长的 MEMORY.md 压缩到合理大小，
> 并解决"每条经验教训都变成永久规则"的问题。

---

## 一、旧 MEMORY.md 结构分析（15KB）

```
MEMORY.md 内容分布：
├── 项目基本信息 ............. ~0.5KB（路径/版本/配置/硬件）
├── 工作模式 ................. ~0.5KB（沟通风格/开发策略）
├── 安全检查脚本清单 ......... ~1KB（6个脚本描述）
├── 硬限制清单 ............... ~1KB（并行度/GPU/DB迁移/分层开发顺序/存储分层）
├── 经验1: 规范唯一来源 ....... ~1.5KB（含详细解释和反例）
├── 经验2: 核心信息唯一来源 ... ~1.5KB（含详细解释和反例）
├── 经验3: 会话启动5步流程 ... ~2KB（含详细步骤和错误处理）
├── 经验4: P0代码审查 ......... ~2.5KB（含启动协议详细描述）
├── 经验5: AI行为红线5条 ...... ~2KB（含每条详细说明）
├── 经验6: 验证无残留6条 ....... ~1.5KB（含详细流程）
├── 经验7: 确认类操作4条 ....... ~1KB（含详细流程）
└── 经验8: 需求来源铁律4条 ..... ~1KB（含详细说明）
```

### 核心问题

| 问题 | 具体表现 |
|------|---------|
| **信息混杂** | 项目信息+工作流程+经验教训+执行规则全在一个文件 |
| **只增不减** | 每条经验都是追加，没有淘汰机制 |
| **解释过度** | 每条经验都有"为什么"和"反例"，占据大量空间 |
| **重复存储** | AGENTS.md已有规则，MEMORY.md又写一遍 |
| **缺乏结构** | 没有分层，阅读者必须读完15KB才知道哪些是当前有效的 |

## 二、压缩策略：分层 + 分离 + 精简

### 策略1：拆分到不同文件（解决信息混杂）

```
MEMORY.md 15KB
    ├── 项目基本信息 (~0.5KB) → CONTEXT.md #Project
    ├── 工作模式 (~0.5KB) → 删除（融入AI默认行为，不需记录）
    ├── 安全检查脚本 (~1KB) → scripts/verify.sh（脚本化）
    ├── 硬限制 (~1KB) → CONTEXT.md #Constraints（仅保留3-5条最高频）
    ├── 经验1-2（规范唯一来源）→ META.md MR1/MR2（升级为元规则）
    ├── 经验3（启动5步）→ playbook/start.md（精简为3步）
    ├── 经验4（P0代码审查）→ playbook/incident.md（收敛）
    ├── 经验5（AI红线5条）→ META.md R4/R5/R6（精简为3条红线）
    ├── 经验6（验证无残留）→ scripts/verify.sh（自动化）
    ├── 经验7（确认止损）→ META.md R7/R8（精简为2条）
    └── 经验8（需求来源）→ playbook/develop.md R10（精简为1条）
```

### 策略2：删除"为什么"（解决解释过度）

**旧格式（每条经验都有解释）**：
```markdown
## 经验1: 规范唯一来源

**原因**: 因为过去出现过多个AGENTS.md副本，
导致AI读取了过时的规则...

**反例**: 2024-01-05，AI读取了根目录的AGENTS.md，
而没有读取 projects/video_management/AGENTS.md，
导致违反了项目特定规则...

**执行要求**: 
1. 所有AI交互必须通过项目根目录的AGENTS.md
2. 禁止在子目录创建AGENTS.md
3. 发现多个AGENTS.md立即报告
```

**新格式（ playbook 中的规则只保留"做什么"）**：
```markdown
R11. 一个规则只存在于一个地方（META.md是唯一规则来源）
```

> "为什么"和"反例"只存在于 incident.md 的经验库中，作为历史参考。
> 执行手册只保留操作指令。

### 策略3：经验≠规则（解决只增不减）

**旧机制**：经验教训 → 自动升级为强制执行规则 → 永久追加

**新机制**：
```
经验发生 → 写入 incident.md 经验库 → 每月Review → 
  ├── 决定升级 → 经过三问检查 → 试运行2周 → 审批 → 
  │   替换 META.md 旧规则 → 生效
  └── 决定不升级 → 保留在经验库 → 作为历史参考
```

## 三、压缩前后对比

### CONTEXT.md — 项目信息（原MEMORY.md前半部分）

| 维度 | 旧 MEMORY.md | 新 CONTEXT.md |
|------|-------------|---------------|
| 项目基本信息 | ~0.5KB（含硬件细节） | ~0.3KB（仅核心属性表格） |
| 硬限制 | ~1KB（6条详细描述） | ~0.3KB（仅3-5条表格） |
| 工作模式 | ~0.5KB | 删除（融入AI行为） |
| 安全检查脚本 | ~1KB（6个脚本描述） | 删除（脚本化到verify.sh） |
| **小计** | **~3KB** | **~0.6KB** |

### META.md — 规则（原MEMORY.md + AGENTS.md）

| 维度 | 旧文件 | 新 META.md |
|------|--------|-----------|
| AGENTS.md 3条红线 | ~0.5KB | → R4/R5/R6（3行） |
| AGENTS.md 2个协议 | ~1KB | → 脚本化/融入playbook |
| MEMORY.md 8条经验 | ~12KB（含解释） | → 15条执行规则（无解释） |
| 元规则 | 无 | 新增5条 |
| **小计** | **~13.5KB** | **~2KB** |

### playbook/*.md — 场景手册（分散的经验流程）

| 维度 | 旧 MEMORY.md | 新 playbook/ |
|------|-------------|-------------|
| 启动5步流程 | ~2KB | start.md ~0.5KB（3步） |
| P0代码审查 | ~2.5KB | incident.md ~0.5KB（收敛） |
| AI行为红线 | ~2KB | develop.md ~0.5KB（速查表） |
| 验证无残留 | ~1.5KB | scripts/verify.sh（自动化） |
| 需求来源 | ~1KB | develop.md ~0.3KB（标记表） |
| **小计** | **~9KB** | **~2.5KB** |

### 总大小对比

```
旧架构总规范大小:
  MEMORY.md .......... 15,561 B
  AGENTS.md .......... 3,205 B
  TODO.md ............ 20,277 B
  P0_*.md × 6 ........ ~55,000 B
  ---------------------------
  合计:               ~94 KB

新架构总规范大小:
  META.md ............ ~2,000 B
  CONTEXT.md ......... ~600 B
  SKILLS.md .......... ~800 B
  playbook/ × 4 ...... ~3,500 B
  scripts/ × 3 ....... ~2,000 B
  ---------------------------
  合计:               ~9 KB

压缩比: 94 KB → 9 KB = 90% 缩减
```

## 四、防增长机制

### 机制1：硬约束计数器

```
CONTEXT.md 顶部:

# CONTEXT — 项目快照
# 大小上限: 1KB | 当前: 0.6KB | 状态: ✅

当大小 > 0.9KB 时:
  → scripts/pre-flight.sh 自动告警
  → 触发"上下文清理"流程
```

### 机制2：约束上限

```
CONTEXT.md #Constraints 最多5条。
新增约束必须淘汰旧约束。
```

### 机制3：外部化

```
旧: TODO.md（20KB，持续增长）
新: 外部工具（GitHub Issues/Linear）

CONTEXT.md 只保留:
#Session
目标: 修复xxx（来源issue#42）
```

### 机制4：月度Review

```
每月检查:
- CONTEXT.md 是否 < 1KB？
- #Constraints 是否 ≤ 5条？
- 是否有未使用的约束？
- 是否可以从脚本获取的信息？（→ 删除，脚本提供）
```

## 五、经验升级的守门机制

### 问题
旧架构：每条经验自动变成规则 → 规则无限增长

### 新机制：三道闸门

```
经验写入 incident.md
        │
        ▼
  ┌─────────────┐
  │ 第一道: 30天冷却 │ ← 经验必须存在30天才可提议升级
  └──────┬──────┘
         │
         ▼
  ┌─────────────┐
  │ 第二道: 三问检查 │ ← 必须全部回答"是"
  │ Q1: 重复问题？    │
  │ Q2: 不能脚本化？  │
  │ Q3: 有旧规则淘汰？│
  └──────┬──────┘
         │
         ▼
  ┌─────────────┐
  │ 第三道: 试运行  │ ← 作为建议运行2周
  └──────┬──────┘
         │
         ▼
  ┌─────────────┐
  │ 第四道: 审批   │ ← 人工确认
  └──────┬──────┘
         │
         ▼
   写入 META.md，淘汰旧规则
```

### 三问检查表模板

```
规则提案: [描述]
提议人: [name]
日期: [date]

Q1: 这条规则解决的是重复出现的问题吗？
    回答: [是/否]  证据: [incident#N 或其他]
    
Q2: 能否用脚本自动化而非规则？
    回答: [能/不能]  说明: [为什么不能用脚本]
    
Q3: 淘汰哪条旧规则？
    回答: [规则编号]  说明: [为什么这条可以淘汰]

审批结果: [通过/拒绝]
审批人: [name]
日期: [date]
```

## 六、总结

| 问题 | 解决方案 | 效果 |
|------|---------|------|
| MEMORY.md 15KB | 拆分+精简+脚本化 | 900B（CONTEXT.md）|
| 规则无限增长 | MR5 + 四道闸门 | 规则≤15条 |
| 经验自动变规则 | 经验库与规则分离 | 只有审批后才能升级 |
| 重复存储 | 单一来源 MR1 | 规则只存在于 META.md |
| P0文件 proliferation | 收敛到 incident.md | 单一文件 |
| 解释过度 | playbook只保留"做什么" | 每条规则1行 |
| TODO.md 20KB | 外部化+CONTEXT.md只存当前 | 删除 |
